Pi Coding Agent 架构分析 —— 如何组合 ai 和 agent 模块

分析 packages/coding-agent 如何利用 packages/ai（模型抽象）和 packages/agent（通用 agent 框架）构建出完整的 coding agent 产品。核心关注：哪些类和接口被复用、哪些能力是 coding-agent 自己扩展的、整体分层关系。

1. 包依赖关系

三个包按依赖方向从底层到上层排列：

pi-ai             (packages/ai)         — 纯模型抽象层，provider-agnostic
  ↑
pi-agent-core     (packages/agent)      — 通用 agent 循环框架，provider-agnostic
  ↑
pi-coding-agent   (packages/coding-agent) — 具体 coding agent 产品

coding-agent 的 package.json 中同时依赖了 @earendil-works/pi-ai 和 @earendil-works/pi-agent-core（注意 agent 包的 npm 名称是 pi-agent-core，目录是 packages/agent）。

3. 从 pi-agent-core 使用的类型和类

Agent 类 —— 最核心的复用

Agent 是 agent-core 暴露的有状态 agent 循环包装器，coding-agent 在 createAgentSession 中直接实例化：

import { Agent } from "@earendil-works/pi-agent-core";

agent = new Agent({
    initialState: { systemPrompt, model, thinkingLevel, tools },
    convertToLlm: convertToLlmWithBlockImages,
    streamFn: async (model, context, options) => modelRuntime.streamSimple(model, context, options),
    onPayload: ...,
    onResponse: ...,
    transformContext: ...,
    steeringMode: ...,
    followUpMode: ...,
    thinkingBudgets: ...,
    transport: ...,
});

Agent 暴露的控制面

成员 / 概念类型coding-agent 如何使用
agent.stateAgentState直接读写 systemPrompt、messages、tools、model、thinkingLevel
agent.prompt()方法启动 agent 运行（发送 user message）
agent.continue()方法在 post-agent-run 处理后继续
agent.steer()方法运行中注入 steering 消息
agent.followUp()方法运行中注入 follow-up 消息
agent.subscribe()(listener) => unsubscribe监听全部 agent 事件（agent_start、turn_start、message_end 等）
agent.abort()方法中止当前运行
agent.beforeToolCallhook 属性coding-agent 注入扩展系统的 tool_call 拦截
agent.afterToolCallhook 属性coding-agent 注入扩展系统的 tool_result 拦截
agent.prepareNextTurnWithContexthook 属性coding-agent 注入 system prompt 和工具刷新逻辑
agent.convertToLlm属性coding-agent 覆盖为带图片过滤的版本
agent.hasQueuedMessages()方法检测 agent 内部队列是否还有待处理消息（决定是否 continue）

基础类型

类型说明
AgentStateagent 状态：systemPrompt、model、thinkingLevel、tools、messages、isStreaming 等
AgentMessage对话消息联合类型 = Message | CustomAgentMessages[keyof CustomAgentMessages]
AgentTool<TParameters, TDetails>工具接口：name、description、parameters (TSchema)、label、execute、executionMode
AgentEventagent 生命周期事件：agent_start/end、turn_start/end、message_start/update/end、tool_execution_*
ThinkingLevel推理等级："off" | "minimal" | "low" | "medium" | "high" | "xhigh" | "max"
StreamFn流函数签名：(model, context, options?) => AssistantMessageEventStream

声明合并

coding-agent 通过 TypeScript declaration merging 扩展 agent-core 的 CustomAgentMessages 接口：

// packages/coding-agent/src/core/messages.ts
declare module "@earendil-works/pi-agent-core" {
    interface CustomAgentMessages {
        bashExecution: BashExecutionMessage;
        compactionSummary: CompactionSummaryMessage;
        branchSummary: BranchSummaryMessage;
        custom: CustomMessage;
        skillInvocation: SkillInvocationMessage;
    }
}

4. coding-agent 的核心扩展

4.1 AgentSession —— 核心编排层

文件：core/agent-session.ts（3328 行）

AgentSession 包裹一个 Agent 实例，在 agent-core 的通用循环之上叠加 coding-agent 专属能力：

能力实现方式
会话持久化通过 SessionManager 将 agent 消息写入 JSONL 文件；监听 message_end 事件自动保存
上下文压缩 (compaction)当 token 超限时触发；从 agent 状态提取消息、调用模型生成摘要、用摘要替换历史消息
分支摘要 (branch summarization)session fork / 分支切换时生成摘要，帮助模型理解历史上下文
自动重试根据 RetrySettings 对可重试错误延迟后重新运行
Bash 执行executeBash 方法，执行后追加 BashExecutionMessage 到 agent state
模型 / thinking 切换setModel、cycleModel、setThinkingLevel、cycleThinkingLevel —— 直接修改 agent.state
工具注册表维护 _toolRegistry（AgentTool）、_toolDefinitions（ToolDefinition）、_toolPromptSnippets、_toolPromptGuidelines
系统 prompt 管理_rebuildSystemPrompt —— 根据活跃工具集、skills、context files 动态重建
队列管理steering / followUp 消息队列，通过 agent 的 queue API 注入（agent.steer() / agent.followUp()）
扩展事件桥接将 agent-core 事件映射到 coding-agent 扩展事件（turn_start → TurnStartEvent 等）

4.2 扩展系统 (Extension System)

目录：core/extensions/

整套插件系统暴露 ~50 种自定义事件类型，核心组件：

  • ExtensionRunner：扩展加载、生命周期管理、事件分发
  • ExtensionAPI / ExtensionContext：给扩展暴露的 API（pi.registerCommand、pi.sendMessage、pi.registerTool、pi.ui.*、pi.session.*、pi.exec.* 等）
  • wrapRegisteredTools：将扩展注册的 tool 包装为带 sourceInfo 的 ToolDefinition
  • 事件类型：BeforeAgentStartEvent、BeforeProviderHeadersEvent、ToolCallEvent、SessionBeforeCompactEvent、InputEvent、ContextEvent 等

4.3 工具系统

目录：core/tools/

7 个内置工具，每个实现 AgentTool<TParameters, TDetails> 接口：

工具名文件功能
readread.ts读取文件（支持图片、行范围、符号查询）
bashbash.ts执行 shell 命令
editedit.ts编辑文件（set_line、replace_lines、insert_after、replace）
writewrite.ts创建或覆写文件
grepgrep.ts文本搜索
find / lsfind.ts、ls.ts文件查找与列表

工厂函数 createCodingTools() / createReadOnlyTools() 组合这些工具并按需注入自定义 cwd 和 operations。

4.4 ModelRuntime —— 模型与认证运行时

文件：core/model-runtime.ts

在 pi-ai 的 createModels + lazy provider 注册之上加入：

  • 内置 provider 目录（@earendil-works/pi-ai/providers/all）
  • 扩展注册的 providers（validateExtensionProvider）
  • 认证存储抽象（AuthStorage / CredentialStore）
  • 自定义模型配置叠加（composeModelProvider）
  • 远程模型目录（withRemoteCatalog）
  • snapshot 查询（可用模型列表、已配置 provider 列表、认证状态）

4.5 SessionManager —— 会话持久化

文件：core/session-manager.ts

基于 JSONL 文件的多分支会话持久化系统：

  • 支持分支（branch）/ 树导航
  • 条目类型：message、compaction、model change、thinking level change、custom message 等
  • JSONL 格式，支持跨版本迁移（migrateSessionEntries）
  • 从 session 文件重建 AgentMessage 数组（buildSessionContext）

4.6 运行模式

目录：modes/

模式文件用途
Interactivemodes/interactive/interactive-mode.ts（6000+ 行）TUI 全功能交互模式
Printmodes/print-mode.ts单次 prompt → 输出（pi -p "..."）
RPCmodes/rpc/通过 JSON-RPC 暴露给外部 IDE 插件

所有三种模式共享同一个 AgentSession 实例，差异仅在 I/O 层（TUI 组件树 vs. stdout 输出 vs. JSON-RPC 协议）。

6. 核心设计模式

组合而非继承

coding-agent 不继承 Agent，而是通过 组合 使用：

  • AgentSession 持有 agent: Agent 实例
  • 通过 agent.subscribe() 监听事件（内部处理 session 持久化、compaction 检查、扩展桥接）
  • 通过 agent.state 读写状态（model、messages、tools、thinkingLevel）
  • 通过 agent.prompt() / agent.steer() / agent.followUp() 注入消息
  • 通过 hook 属性注入自定义行为（beforeToolCall、afterToolCall、prepareNextTurnWithContext）

Provider-Agnostic 设计

pi-agent-core 通过 StreamFn 类型保持与具体 provider 解耦。coding-agent 在初始化时将 pi-ai/compat 的 streamSimple 注入：

// sdk.ts
setDefaultStreamFn(streamSimple); // 注入默认流实现

agent-core 本身不 import pi-ai/compat，由上层 coding-agent 完成注入。

声明合并扩展消息类型

agent-core 的 AgentMessage 通过 CustomAgentMessages 接口留出扩展点。coding-agent 使用 declaration merging 注入 5 种专属消息类型（BashExecutionMessage、CompactionSummaryMessage 等），这些消息在 agent 循环中被当作一等公民处理。

事件驱动的双层监听

agent-core 发出 AgentEvent，AgentSession._handleAgentEvent 作为内部处理器做三件事：

  1. 桥接到扩展系统（_emitExtensionEvent）
  2. 转发给模式层（_emit → I/O 层 listener）
  3. 执行内部逻辑（session 持久化、compaction 检测、retry 检测）

7. 总结

层次包职责提供什么
L0 模型层 pi-ai 纯数据与流抽象 Model、Provider、Api、Message、Usage、streamSimple、CredentialStore、工具函数
L1 Agent 框架 pi-agent-core 通用 agent 循环 Agent、AgentState、AgentTool、AgentEvent、StreamFn、ThinkingLevel、队列管理
L2 Coding 编排 coding-agent/core Coding 专用逻辑 AgentSession、compaction、retry、bash execution、tool registry、system prompt、扩展系统、session 持久化、model runtime
L3 I/O 适配 coding-agent/modes 用户交互 InteractiveMode (TUI)、PrintMode (stdout)、RpcClient (JSON-RPC)

coding-agent 的核心价值在于 L2 编排层：它不重新发明 agent 循环或模型抽象，而是复用 Agent + pi-ai 的基础能力，通过组合、hook 注入和声明合并将通用 agent 框架转化为一个完整的 coding agent 产品。
