Pi AI 抽象层架构分析

本文分析 packages/ai 包的设计，它如何将不同 LLM 提供商（Anthropic、OpenAI、Google、DeepSeek 等）的差异抽象为统一的编程接口。

整体定位

packages/ai 是 pi 的 LLM 抽象层，核心职责是屏蔽不同 LLM 提供商在协议、数据格式、能力上的差异，向消费方（coding-agent、tui、extension）提供一套统一的数据类型、流协议和调用接口。

coding-agent / tui / extension  ←  消费方
        |
   packages/ai                    ←  统一抽象层
        |
   Anthropic / OpenAI / Google / ...  ←  后端 API

1. ToolMessage 统一

差异点

不同 API 对工具调用（tool call / tool result）的 wire format、ID 约束和消息角色完全不同。

维度Anthropic MessagesOpenAI ResponsesOpenAI Chat Completions
tool call 格式content_block 中 type: "tool_use", id: "toolu_xxx", input: {}response.output_item 中 type: "function_call", call_id + id 分离choices[0].delta.tool_calls 中 function.name + arguments
tool result 角色role: "user"，content: [{type: "tool_result", ...}]type: "function_call_output", call_id 匹配role: "tool", tool_call_id 匹配
ID 约束^[a-zA-Z0-9_-]+$，最长 64 字符pipe 分隔 call_id|item_id，item_id 可 450+ 字符最长 40 字符
流式 tool callinput_json_delta 推送 partial JSONresponse.function_call_arguments.deltadelta.tool_calls[].function.arguments

统一方式：三层归一

第一层：统一数据类型

export interface ToolCall {
    type: "toolCall";
    id: string;              // 归一化后的 ID
    name: string;
    arguments: Record<string, any>;
    thoughtSignature?: string;  // Google 特有，跨模型时丢弃
}

export interface ToolResultMessage {
    role: "toolResult";
    toolCallId: string;      // 与 ToolCall.id 匹配
    toolName: string;
    content: (TextContent | ImageContent)[];
    isError: boolean;
    addedToolNames?: string[];  // 延迟加载工具
}

第二层：统一的 ID 归一化

每个 API 适配器提供自己的 normalizeToolCallId，通过 transformMessages() 的回调注入。例如：

  • openai-responses.ts 的 normalizeToolCallId：处理 call_id|item_id 格式，跨模型时重建 item ID
  • anthropic-messages.ts 的 normalizeToolCallId：将 [^a-zA-Z0-9_-] 替换为 _，截断到 64 字符
  • openai-completions.ts 的 normalizeToolCallId：管道 ID 截断到 40 字符，必要时哈希去重

transformMessages() 预处理消息列表时，将 AssistantMessage 中的 tool call ID 逐一归一化，同时为后续 ToolResultMessage 建立反向映射。

第三层：消息格式转换

每个适配器的 convertMessages 将统一数据模型转为该后端原生格式：

  • Anthropic：toolResult 转为 role: "user" 的 tool_result content block，支持 tool_reference 延迟加载
  • OpenAI Responses：toolResult 转为 function_call_output item，支持 custom_tool_call 和 tool_search_call
  • OpenAI Chat Completions：toolResult 转为 role: "tool" 消息

2. LLM Streaming 统一

差异点

各 API 的 SSE 事件格式完全不同。

API流式事件增量字段
Anthropicmessage_start, content_block_start, content_block_delta, message_deltatext_delta, thinking_delta, input_json_delta, signature_delta
OpenAI Responsesresponse.output_text.delta, response.function_call_arguments.delta, response.reasoning_text.delta各 delta 独立事件
OpenAI Chat Completionschoices[0].delta.content, choices[0].delta.tool_calls, choices[0].delta.reasoning_content单个 delta 对象多字段

统一方式

统一事件协议

export type AssistantMessageEvent =
    | { type: "start"; partial: AssistantMessage }
    | { type: "text_start"; contentIndex: number; partial: AssistantMessage }
    | { type: "text_delta"; contentIndex: number; delta: string; partial: AssistantMessage }
    | { type: "text_end"; contentIndex: number; content: string; partial: AssistantMessage }
    | { type: "thinking_start"; contentIndex: number; partial: AssistantMessage }
    | { type: "thinking_delta"; contentIndex: number; delta: string; partial: AssistantMessage }
    | { type: "thinking_end"; contentIndex: number; content: string; partial: AssistantMessage }
    | { type: "toolcall_start"; contentIndex: number; partial: AssistantMessage }
    | { type: "toolcall_delta"; contentIndex: number; delta: string; partial: AssistantMessage }
    | { type: "toolcall_end"; contentIndex: number; toolCall: ToolCall; partial: AssistantMessage }
    | { type: "done"; reason: ...; message: AssistantMessage }
    | { type: "error"; reason: ...; error: AssistantMessage }

统一流接口

export class AssistantMessageEventStream extends EventStream<AssistantMessageEvent, AssistantMessage> {

统一函数签名

所有适配器模块都导出两个同名函数：

export const stream: StreamFunction<TApi, TOptions> = (model, context, options?) => AssistantMessageEventStream;
export const streamSimple: StreamFunction<TApi, SimpleStreamOptions> = (model, context, options?) => AssistantMessageEventStream;

ProviderStreams 接口将这两个函数绑定在一起：

export interface ProviderStreams {
    stream(model, context, options?): AssistantMessageEventStream;
    streamSimple(model, context, options?): AssistantMessageEventStream;
}

每个适配器内部将后端原生事件映射到统一事件：

// 伪代码：所有适配器的共同模式
export const stream = (model, context, options) => {
    const stream = new AssistantMessageEventStream();
    (async () => {
        try {
            stream.push({ type: "start", partial: initialOutput });
            for await (const rawEvent of rawApiStream) {
                // 将 rawEvent 映射为统一的 thinking_start/text_delta/toolcall_end 等
                stream.push(unifiedEvent);
            }
            stream.push({ type: "done", ... });
            stream.end();
        } catch (e) {
            stream.push({ type: "error", ... });
            stream.end();
        }
    })();
    return stream;
};

3. Thinking/Reasoning 统一

差异点

不同提供商对推理（thinking/reasoning）的参数名、控制值、启用方式完全不同。

API参数值备注
Anthropic (adaptive)thinking.type: "adaptive" + output_config.effortlow|medium|high|xhigh|maxClaude 4.6+
Anthropic (legacy)thinking.type: "enabled" + budget_tokenstoken 数Claude 3.x/4.x
OpenAI Responsesreasoning.effortlow|medium|high|xhigh|max
DeepSeekthinking.type + reasoning_effortbinary + optional effort
Qwenenable_thinking: booleanbinary
OpenRouterreasoning.effort归一化 effort
Togetherreasoning.enabled + reasoning_effortbinary + effort
z.aithinking.typebinary
llama.cpp 等reasoning_content 响应字段仅响应，无请求参数
Chat 模板chat_template_kwargs.*可配置变量通过 $var 引用

统一方式：三层映射

第一层：统一的 reasoning 层级枚举

export type ThinkingLevel = "minimal" | "low" | "medium" | "high" | "xhigh" | "max";
export type ModelThinkingLevel = "off" | ThinkingLevel;

消费方只需传 SimpleStreamOptions.reasoning?: ThinkingLevel。

第二层：thinkingLevelMap 模型级映射

每个 Model 可包含 thinkingLevelMap，将 pi 的统一层级映射到提供商特定值：

export type ThinkingLevelMap = Partial<Record<ModelThinkingLevel, string | null>>;

// 示例：OpenAI o3
{ off: "none", minimal: "low", low: "low", medium: "medium", high: "high", xhigh: "high", max: "max" }

// 示例：DeepSeek
{ off: null }  // null = 不支持关闭

clampThinkingLevel() 根据模型的可用层级将请求层级约束到最近的已支持值。

第三层：thinkingFormat 策略分发

在 openai-completions.ts 的 buildParams() 中，根据模型配置的 compat.thinkingFormat 分支到不同逻辑，支持 10+ 格式变体：

if (compat.thinkingFormat === "zai" && model.reasoning) {
    params.thinking = reasoningEffort ? { type: "enabled" } : { type: "disabled" };
} else if (compat.thinkingFormat === "deepseek" && model.reasoning) {
    params.thinking = reasoningEffort ? { type: "enabled" } : { type: "disabled" };
    if (reasoningEffort) params.reasoning_effort = mappedEffort;
} else if (compat.thinkingFormat === "openrouter" && model.reasoning) {
    params.reasoning = { effort: mappedEffort ?? "none" };
} else if (/* ... */) {
    // ... 各分支
}

响应端统一

无论后端如何返回推理内容，所有适配器都转为统一的 ThinkingContent：

{ type: "thinking", thinking: "...", thinkingSignature: JSON.stringify(original), redacted?: boolean }

并发射 thinking_start / thinking_delta / thinking_end 事件。

4. Prompt Cache 统一

差异点

提供商机制粒度长缓存
Anthropiccache_control: { type: "ephemeral" } 标记到 content blockcontent block 级ttl: "1h"
OpenAI Responsesprompt_cache_key + prompt_cache_retention整个输入级"24h"
OpenAI Chat Completionsprompt_cache_key + prompt_cache_retention整个输入级"24h"
Anthropic-compatible 第三方cache_control 标记content block 级可选 1h
OpenAI 第三方（Fireworks 等）prompt_cache_key + session affinity整个输入 + 路由

统一方式

统一开关

export type CacheRetention = "none" | "short" | "long";

各适配器从 options.cacheRetention 读取偏好，映射到本地语义。

Anthropic 风格实现

在 anthropic-messages.ts 中，cacheControl 按三条规则注入：

  1. System prompt 的 text content block 加 cache_control
  2. 最后一条工具定义（tools 数组最后一个）加 cache_control
  3. 最后一条 user/assistant/tool-result 消息的最后一个 text block 加 cache_control

long 时加 ttl: "1h"。

OpenAI 风格实现

// openai-completions.ts / openai-responses.ts
params.prompt_cache_key = clampOpenAIPromptCacheKey(options?.sessionId);
params.prompt_cache_retention = cacheRetention === "long" ? "24h" : undefined;

clampOpenAIPromptCacheKey 将 sessionId 裁剪到 64 字符。

交叉兼容

有些 OpenAI 兼容提供商采用 Anthropic 的 cache_control 方式。openai-completions.ts 的 applyAnthropicCacheControl() 在参数构建后，按 Anthropic 规则为已构建的 OpenAI 格式消息注入 cache_control。

Session affinity

部分提供商需要请求路由到同一副本。统一的 sessionId 通过不同 header 发送：

  • OpenAI: session_id + x-client-request-id
  • OpenRouter: x-session-id
  • Fireworks Anthropic: x-session-affinity
  • 通过 sendSessionAffinityHeaders 配置控制

使用统计统一

Usage 统一包含缓存字段：

export interface Usage {
    cacheRead: number;
    cacheWrite: number;
    cacheWrite1h?: number;  // Anthropic 独有，按 2x 计费
}

各适配器从后端原生字段提取：

后端cacheReadcacheWrite
Anthropicusage.cache_read_input_tokensusage.cache_creation_input_tokens

Adapter 模式详解

每个 API 适配器是一个独立模块，通过 lazyApi() 延迟加载，通过 createProvider() 注册到模型系统。

@earendil-works/pi-ai
├── api/
│   ├── anthropic-messages.ts     ← Anthropic 适配器（含自有 SSE parser）
│   ├── openai-responses.ts       ← OpenAI Responses 适配器
│   ├── openai-completions.ts     ← OpenAI Chat Completions 适配器
│   ├── bedrock-converse-stream.ts
│   ├── google-generative-ai.ts
│   ├── google-vertex.ts
│   ├── mistral-conversations.ts
│   ├── azure-openai-responses.ts
│   ├── openai-codex-responses.ts
│   ├── pi-messages.ts
│   ├── cloudflare.ts
│   ├── lazy.ts                   ← lazyApi() 动态加载包装器
│   ├── transform-messages.ts     ← 共享消息预处理
│   ├── simple-options.ts         ← 通用选项构造
│   ├── constrained-sampling.ts   ← JSON schema / grammar 约束采样
│   └── openai-prompt-cache.ts    ← cache key 裁剪
├── providers/
│   ├── anthropic.ts              ← Anthropic 提供商标记（模型目录 + API 绑定）
│   ├── openai.ts
│   ├── deepseek.ts
│   └── ...（每个提供商一个文件）
└── types.ts                     ← 统一数据模型

适配器契约：每个 API 模块必须导出 stream 和 streamSimple（符合 ProviderStreams 接口）。lazyApi() 包装器确保模块在首次调用时才加载，import 缓存去重负载。

提供商标记：每个 providers/*.ts 文件通过 createProvider() 注册，绑定：ID、名称、baseUrl、认证方式、模型目录、API 实现。

总结

packages/ai 通过四层抽象解决 LLM 提供商差异：

抽象层面解决什么问题核心机制
统一数据类型各 API 的消息/工具/用量格式不同AssistantMessage, ToolCall, Usage 等收敛类型
统一流协议各 API 的 SSE 事件不同AssistantMessageEventStream + 12 种事件类型
统一 thinking 映射thinking 参数名/值/启用方式各异thinkingLevelMap + thinkingFormat 策略 + 层级枚举
统一缓存控制缓存机制/粒度/时长不同CacheRetention + 双风格（Anthropic 标记 / OpenAI prompt key）
